# Spec - 누락 보강 및 데이터 신뢰성 정책

## 1. 목적

Yeonsik's Note는 자기관리 및 계발 기록 앱이다. 따라서 기록은 단순히 많이 입력되는 것보다, 언제 입력되었고 어떤 신뢰도를 가지는지가 중요하다.

이 명세의 목적은 지난 날짜에 뒤늦게 추가되는 기록을 일반 기록과 구분하고, 통계 신뢰도를 훼손하지 않도록 입력 정책, 라벨, 통계 반영 기준을 정의하는 것이다.

## 2. 핵심 원칙

지난 날짜에는 일반적인 빠른 추가를 허용하지 않는다.

단, 실제로 수행했지만 입력을 놓친 기록은 명시적인 `누락 보강` 흐름으로만 추가할 수 있다.

```text
오늘 기록 = 일반 기록
지난 날짜에 나중에 추가한 기록 = 누락 보강 기록
삭제 = 항상 tombstone soft delete
```

`누락 보강`은 사용자의 실수를 보정하기 위한 기능이지, 과거 데이터를 마음대로 재작성하기 위한 기능이 아니다.

## 3. 용어

| 용어 | 의미 |
|---|---|
| 기록 날짜 | 운동/식사/체중/할 일이 실제로 속하는 날짜 |
| 입력 시각 | 사용자가 앱에 데이터를 저장한 실제 시각 |
| 당일 기록 | 기록 날짜와 입력 날짜가 같은 기록 |
| 누락 보강 | 지난 날짜에 뒤늦게 추가한 기록 |
| tombstone 삭제 | hard delete 없이 `deletedAt`을 남기는 삭제 방식 |
| 행동 통계 | 실제 수행량, 체중 추이, 칼로리 평균처럼 기록 날짜 기준으로 보는 통계 |
| 기록 습관 통계 | 당일 기록률, 연속 기록일, 체크인 성실도처럼 입력 시점 기준으로 보는 통계 |

## 4. 입력 정책

### 4.1 오늘 날짜

오늘 날짜는 일반 입력을 허용한다.

허용 작업:

```text
빠른 작업
Quick Capture
할 일 추가/수정/완료
메모 추가/수정
운동/식사/체중 추가/수정/삭제
```

### 4.2 지난 날짜

지난 날짜는 일반 빠른 추가를 막는다.

기본 정책:

```text
빠른 작업 버튼은 숨기거나 비활성화한다.
일반 추가 버튼은 숨긴다.
기존 항목 삭제는 허용한다.
기존 항목 수정은 정책별로 제한한다.
새 항목 추가는 누락 보강 버튼을 통해서만 허용한다.
```

권장 UI 문구:

```text
누락 보강
지난 기록 보강
정정 입력
```

지난 날짜에 새 항목을 추가하려면 사용자가 명시적으로 `누락 보강`을 선택해야 한다.

권장 확인 문구:

```text
지난 날짜에 기록을 추가합니다. 이 항목은 누락 보강으로 표시됩니다.
```

### 4.3 미래 날짜

미래 날짜는 실제 수행 기록을 추가하지 않는다.

허용 범위:

```text
할 일 예약
일정성 메모
```

운동/식사/체중처럼 실제 수행 후 기록해야 하는 데이터는 미래 날짜에 일반 기록으로 추가하지 않는다.

## 5. Quick Action과 Quick Capture 정책

### 5.1 빠른 작업

빠른 작업은 선택된 날짜의 문맥 안에서 사용하는 기능이다.

정책:

```text
오늘 또는 허용된 날짜에서만 일반 빠른 작업을 연다.
지난 날짜에서는 기본적으로 빠른 작업을 열지 않는다.
지난 날짜에 필요한 추가는 누락 보강 흐름으로 분리한다.
```

이 정책은 과거 날짜를 보고 있다가 실수로 새 기록을 추가하는 문제를 줄인다.

### 5.2 Quick Capture

Quick Capture는 전역 즉시 입력 기능이다.

정책:

```text
항상 오늘 날짜 기준으로 저장한다.
선택된 캘린더 날짜의 영향을 받지 않는다.
과거 날짜 누락 보강 용도로 사용하지 않는다.
```

Quick Capture는 속도 우선 기능이고, 누락 보강은 신뢰도 우선 기능이다. 두 흐름은 합치지 않는다.

## 6. 데이터 모델 정책

현재 엔티티는 `updatedAt`, `deletedAt`, `deviceId`를 공통으로 가진다. 데이터 신뢰도를 정확히 관리하려면 입력 시각과 기록 날짜를 분리해서 보존해야 한다.

권장 확장 필드:

```ts
type BackfillMetadata = {
  isBackfilled: boolean;
  backfilledAt: string | null;
  backfillReason?: string | null;
};
```

장기적으로는 모든 syncable entity에 `createdAt`을 추가하는 것이 좋다.

```ts
type SyncableEntity = {
  id: string;
  createdAt: string;
  updatedAt: string;
  deletedAt: string | null;
  deviceId: string;
};
```

역할 분리:

```text
date / dueDate / recordDate = 기록이 속하는 날짜
createdAt = 실제 입력 시각
updatedAt = 마지막 수정 시각
backfilledAt = 누락 보강으로 입력한 시각
deletedAt = 삭제 tombstone 시각
```

주의:

`updatedAt`은 수정할 때 바뀌므로 입력 시각 판정에 사용하면 안 된다. 기록 습관 통계를 만들려면 `createdAt` 또는 `backfilledAt`이 필요하다.

## 7. 라벨 표시 정책

누락 보강 기록은 UI에서 일반 기록과 구분한다.

권장 라벨:

```text
누락 보강
```

표시 위치:

```text
날짜 상세 목록의 항목 우측
운동/식사/체중 기록 row
Markdown export
통계 카드 보조 문구
공유 이미지 또는 리포트 하단 주석
```

예시:

```text
운동 30분 · 누락 보강
식사 620 kcal · 누락 보강
체중 72.4 kg · 누락 보강
```

통계에 누락 보강이 포함된 경우에는 집계 단위에 표시한다.

```text
이번 주 운동 4회 · 누락 보강 1건 포함
평균 칼로리 1,940 kcal · 누락 보강 포함
```

## 8. 통계 반영 정책

누락 보강 기록은 모든 통계에서 동일하게 처리하지 않는다.

핵심 기준:

```text
실제 행동 통계에는 포함한다.
기록 습관 통계에는 제외한다.
포함된 경우에는 반드시 누락 보강 포함 여부를 표시한다.
```

### 8.1 포함하는 통계

아래 통계는 기록 날짜 기준의 실제 행동 데이터이므로 누락 보강을 포함한다.

| 통계 | 반영 | 이유 |
|---|---:|---|
| 운동 횟수/누적 | 포함 | 실제 수행한 운동량을 과소평가하지 않기 위해 |
| 식사 칼로리/영양소 평균 | 포함 | 실제 섭취 기록이면 늦게 입력되어도 행동 데이터이기 때문 |
| 체중 평균/추이 | 포함 | 실제 측정값이면 입력 시점보다 측정 날짜가 중요하기 때문 |
| 날짜별 기록 목록 | 포함 | 해당 날짜에 속한 기록이므로 사용자가 확인해야 함 |
| export/report | 포함 | 원자료 보존이 목적이므로 제외하지 않음 |

### 8.2 제외하는 통계

아래 통계는 입력 습관을 측정하므로 누락 보강을 제외한다.

| 통계 | 반영 | 이유 |
|---|---:|---|
| 연속 기록일 | 제외 | 나중에 채운 기록으로 streak가 복구되면 의미가 훼손됨 |
| 당일 기록률 | 제외 | 실제 당일에 기록했는지를 보는 지표이기 때문 |
| 오늘 완료율 | 제외 | 오늘의 실행 흐름과 무관한 과거 보정 입력이기 때문 |
| 체크인 성실도 | 제외 | 기록 습관 평가 지표이기 때문 |
| 빠른 기록 사용률 | 제외 | Quick Capture/빠른 작업의 즉시성 지표와 다르기 때문 |

### 8.3 식사 기록의 신뢰도

식사 기록은 기억 기반 입력일 가능성이 높아 오차가 크다.

정책:

```text
식사 누락 보강은 평균/누적 통계에 포함한다.
단, 통계 카드와 export에 누락 보강 포함 여부를 표시한다.
향후 신뢰도 등급을 추가할 수 있다.
```

미래 확장 예시:

```ts
type RecordConfidence = "direct" | "backfilled" | "estimated";
```

## 9. 삭제 정책

누락 보강 기록도 기존 삭제 정책을 그대로 따른다.

```text
hard delete 금지
deletedAt = nowIso()
updatedAt = nowIso()
deviceId = currentDeviceId
```

삭제된 누락 보강 기록은 UI, 통계, export, calendar marker에서 제외한다.

동기화 병합에서 같은 `updatedAt`이면 tombstone이 우선한다.

## 10. 동기화 정책

누락 보강 메타데이터는 local-first snapshot과 Supabase mapper 양쪽에서 보존되어야 한다.

필수 조건:

```text
Supabase가 없어도 로컬에서 동작한다.
동기화가 켜져 있으면 isBackfilled/backfilledAt이 remote에 전파된다.
pull/realtime merge에서 backfill metadata가 사라지면 안 된다.
Tauri 전용 API와 무관한 순수 데이터 정책으로 유지한다.
```

마이그레이션 시 기존 데이터는 다음 기본값을 가진다.

```ts
isBackfilled: false
backfilledAt: null
backfillReason: null
```

## 11. 기록 탭 UX 정책

날짜를 선택했을 때는 먼저 읽기 화면을 보여준다.

오늘 날짜:

```text
빠른 작업 표시
일반 추가 허용
삭제 허용
```

지난 날짜:

```text
빠른 작업 숨김 또는 비활성화
일반 추가 숨김
누락 보강 버튼 표시
삭제 허용
항목에는 누락 보강 라벨 표시
```

빈 상태 문구:

```text
이 날짜에는 아직 기록이 없습니다.
```

지난 날짜 빈 상태에서는 보강 진입점을 분리해서 표시한다.

```text
이 날짜에는 아직 기록이 없습니다.
누락된 기록이 있다면 보강으로 추가할 수 있습니다.
```

## 12. Export 및 공유 정책

Markdown export, 리포트, X 공유 MVP에는 누락 보강 여부를 숨기지 않는다.

예시:

```text
2026-06-08
- 운동: 하체 근력 · 누락 보강
- 식사: 닭가슴살 샐러드, 520 kcal · 누락 보강
```

집계 요약에는 보강 포함 여부를 표시한다.

```text
주간 운동 4회, 식사 기록 12건
누락 보강 2건 포함
```

## 13. 구현 우선순위

### Phase A - UI 정책 잠금

```text
지난 날짜에서 빠른 작업 비활성화
지난 날짜 일반 추가 숨김
누락 보강 버튼 추가
누락 보강 확인 문구 추가
```

### Phase B - 메타데이터 추가

```text
isBackfilled/backfilledAt/backfillReason 추가
localStorage migration 추가
Supabase schema/mapper 추가
recordAggregation selector 보강
```

### Phase C - 통계 분리

```text
행동 통계에는 누락 보강 포함
기록 습관 통계에는 누락 보강 제외
통계 카드에 누락 보강 포함 건수 표시
테스트 추가
```

### Phase D - 리포트/공유 반영

```text
Markdown export 라벨 표시
공유 이미지/텍스트에 누락 보강 포함 표시
필터 옵션 추가 검토
```

## 14. 테스트 기준

필수 테스트:

```text
오늘 날짜는 빠른 작업이 가능하다.
지난 날짜는 일반 빠른 작업이 열리지 않는다.
지난 날짜는 누락 보강으로만 새 기록을 추가할 수 있다.
누락 보강 기록은 라벨이 표시된다.
누락 보강 기록은 운동/식사/체중 통계에 포함된다.
누락 보강 기록은 연속 기록일/당일 기록률에는 포함되지 않는다.
deletedAt !== null인 누락 보강 기록은 모든 UI/통계/export에서 제외된다.
동기화 후에도 누락 보강 메타데이터가 유지된다.
```

권장 pure function 테스트:

```text
isPastLocalDate()
canOpenQuickActionForDate()
createBackfilledRecord()
getBackfillSummary()
calculateHabitStatsExcludingBackfills()
calculateBehaviorStatsIncludingBackfills()
```

## 15. 수용 기준

이 명세가 구현되었다고 판단하려면 다음 조건을 만족해야 한다.

```text
지난 날짜에 일반 빠른 작업으로 새 기록을 추가할 수 없다.
지난 날짜 추가는 누락 보강이라는 명시적 동작을 거친다.
누락 보강 기록은 UI와 export에서 라벨로 구분된다.
행동 통계에는 누락 보강이 포함된다.
기록 습관 통계에는 누락 보강이 제외된다.
통계에 누락 보강이 포함된 경우 사용자에게 표시된다.
deletedAt tombstone 정책은 유지된다.
Windows desktop, Windows web, Android에서 동일한 데이터 정책을 따른다.
Supabase가 없어도 local-first로 동작한다.
```
